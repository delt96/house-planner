# 실측 데이터 로드맵 & 방 실측 UI 설계 문서

- **작성일:** 2026-07-02
- **상태:** 설계 확정 (구현 대기)
- **선행 문서:** `2026-07-01-honsu-list-floor-plan-design.md`

## 1. 배경 / 목적

입주 전 집을 방문해 줄자로 실측한다는 전제에서, 현재 앱에 **담을 곳이 없는 측정값**을 정리하고 단계별로 수용한다.

실측 요구 항목:

1. 콘센트가 벽·문에서 얼마나 떨어져 있는지
2. 창문 폭·높이
3. 문 열리는 방향
4. 문 통과 폭·높이 — **가구가 안 들어가면 치명적이므로 최우선**
5. 천장 높이 — **방마다 다를 수 있음** (방별 데이터)

설계 원칙: *싸면서 실패를 가장 많이 막는 것부터, 구조 변경은 뒤로.*

## 2. 현재 상태 요약

- **반입 판정** (`web/src/carryin.js`): 현관문·엘리베이터 문·엘리베이터 내부 3개 체크만. 실내 방문(안방·드레스룸 문)에서 막히는 경우를 못 잡는다.
- **방** (`rooms`): 이름 + 가로×세로의 맨 사각형. 높이·문·창·콘센트 없음.
- **후보** (`candidates`): 가로×세로×높이 있음 — 비교할 천장·문 데이터가 없어 높이가 활용되지 못함.

## 3. 단계 로드맵

| 단계 | 이름 | 핵심 | 구조 변경 |
|---|---|---|---|
| 1 | 반입 통과 판정 완성 | 실내 방문 통과 체크 | 없음 (컬럼 추가만) |
| 2 | 방별 실측 | 천장 높이 + 문·창·콘센트 (벽 앵커) | `room_features` 신설 |
| 3 | 사다리차/창문 반입 | 정상 동선 불가 시 대체 경로 판정 | 작음 |
| 4 | 배치 현실성 | 도어 스윙·겹침·통행 여유·콘센트 앵커 | 로직 위주 |

각 단계는 독립적으로 자체 계획 → 구현 사이클을 돈다. 이 문서는 **단계 1·2를 구현 가능한 수준으로**, 단계 3·4는 개요 수준으로 확정한다.

## 4. 단계 1 — 반입 통과 판정 완성 (최우선)

### 데이터

```
home_settings + room_door_width_cm   numeric NULL   실내에서 가장 좁은 문의 폭
              + room_door_height_cm  numeric NULL   그 문의 통과 높이
```

"집에서 가장 좁은 실내 문" 하나를 대표값으로 입력한다. 방별 문 데이터(단계 2)가 생겨도 반입 판정은 이 대표값을 계속 쓴다 — 반입 경로(어느 방까지 가는지) 모델링은 과설계이므로 하지 않는다. `room_features`의 문 치수로 대표값을 자동 산출하는 것은 향후 개선.

### 로직

- `carryin.js`의 `evaluateCarryIn`에 `방문` 체크 추가 — 기존 `throughOpening` 재사용, 현관문과 동일 방식.
- 대각선 기울이기 판정은 계속 비범위(보수적 판정 유지). 세워넣기는 기존 정렬 비교(`sortedDims`)로 이미 커버된다.

### UI

- `CarryInPage`에 `방문(실내에서 가장 좁은 문)` fieldset 추가 — 폭·높이 2칸.
- 기존 `CarryInBadge` 사유 문자열에 자동 반영: "방문 통과 불가", "방문 빠듯".

## 5. 단계 2 — 방별 실측 (천장 + 문·창·콘센트)

### 5.1 데이터 모델 — 벽 앵커 방식

창·문·콘센트는 모두 **벽에 붙는 것**이므로 자유 좌표(x,y)가 아니라 `벽 + 모서리로부터 거리`로 담는다. 실측 방식("남쪽 벽, 모서리에서 30cm")과 일치하고, 방을 드래그하면 부착물이 자동으로 따라온다.

```
rooms + ceiling_height_cm  numeric NULL   방별 천장 높이

room_features             방 부착물 (문·창문·콘센트)
  id               serial PK
  room_id          int NOT NULL FK -> rooms(id) ON DELETE CASCADE
  kind             text NOT NULL      'door' | 'window' | 'outlet'
  wall             text NOT NULL      'N' | 'E' | 'S' | 'W'
  offset_cm        numeric NOT NULL   벽 시작 모서리에서 거리 (규약은 아래)
  width_cm         numeric NULL       door/window: 개구부 폭
  height_cm        numeric NULL       door: 통과 높이 / window: 창 높이
  sill_height_cm   numeric NULL       window: 창턱(바닥→창 하단) 높이
  floor_height_cm  numeric NULL       outlet: 바닥에서 높이
  swing            text NULL          door: 'in-left'|'in-right'|'out-left'|'out-right'
  sort_order       int DEFAULT 0
  created_at       timestamptz DEFAULT now()
```

**offset 기준 규약** (평면도 위에서 본 기준):
- N(위)·S(아래) 벽: **서쪽(왼쪽) 모서리**에서 동쪽으로.
- E(오른쪽)·W(왼쪽) 벽: **북쪽(위) 모서리**에서 남쪽으로.
- 즉 항상 "화면에서 위/왼쪽 모서리"가 0점. 벽 길이는 N/S = `width_cm`, E/W = `depth_cm`.

기준점은 **벽 모서리로 통일**한다. "문에서부터 몇 cm" 편의 입력은 문 없는 벽·문 둘인 방에서 성립하지 않으므로 비범위(향후 개선).

### 5.2 입력 UI — 패널 "＋" 흐름

`LayoutPage` 사이드 패널의 방 카드를 확장한다:

```
거실 (450×360) · 천장 [240]cm                [삭제]
 ├ 🚪 문    남쪽 · 모서리 30cm · 폭80 · 높이204 · 안쪽/좌   [수정][삭제]
 ├ ⊞ 창문  북쪽 · 모서리 90cm · 폭180 · 높이120 · 턱90    [수정][삭제]
 ├ ⚡ 콘센트 동쪽 · 모서리 150cm · 바닥 30cm               [수정][삭제]
 └ [＋ 추가 ▾] → 문 / 창문 / 콘센트 3택
```

- **＋를 누르면 종류 3택**, 고르면 그 종류의 필드만 인라인 폼으로 펼쳐진다.
  - 공통 선두 필드: 벽(북/동/남/서 선택) · 모서리에서(cm)
  - 문: 폭 · 통과 높이 · 열림(안/밖 × 좌/우)
  - 창문: 폭 · 높이 · 창턱 높이
  - 콘센트: 바닥에서 높이
- 실측값은 줄자로 잰 숫자를 타이핑하는 것이므로 **폼 입력이 정본**. 캔버스 클릭 배치는 비범위(향후 개선).

### 5.3 평면도 렌더링 — 표준 도면 기호

저장 즉시 SVG 평면도의 해당 벽에 그린다 — "잰 값이 맞나"를 눈으로 즉시 검증하는 것이 이 설계의 핵심 보상.

- **문**: 벽에 개구부(틈) + 90° 부채꼴 호. 호의 방향은 `swing` 값으로 결정 → 열림 방향이 한눈에 보인다.
- **창문**: 창 폭만큼 벽 위 이중선(파란 계열).
- **콘센트**: 벽 위 작은 점(⚡).
- **수직 정보 분리 원칙**: 위에서 본 그림에 높이류(문 통과 높이·창 높이·창턱·콘센트 높이)는 원리적으로 그릴 수 없다. 기호는 위치만 나타내고, 숫자는 **기호 클릭 시 칩**으로 표시 (`W180 · H120 · 턱90`).
- 기호 클릭 ↔ 패널 항목 상호 하이라이트.
- 방 드래그 시 부착물은 벽 앵커라 자동으로 따라온다 (추가 코드 불필요).

### 5.4 검증

- `wall`은 N/E/S/W만, `offset_cm ≥ 0`.
- `offset_cm + (width_cm ?? 0) > 벽 길이`면 400 + 메시지 — 창·문이 벽 밖으로 나가는 입력 실수 차단.
- `kind`별 필수 필드: door/window는 `width_cm` 필수, outlet은 폭 없음.

### 5.5 천장 높이 활용 (단계 2 범위)

- 입력: 방 카드에서 방별로 입력 (방마다 다를 수 있음).
- 판정: 확정 후보의 `height_cm`가 **입력된 천장 중 최저값**을 넘으면 경고. 표시 위치는 `CarryInBadge`가 이미 뜨는 곳(홈 목록·품목 상세)에 경고 칩으로 나란히. 배치 좌표 기반 방별 정밀 비교는 단계 4로 미룬다.

### 5.6 API

```
PATCH  /api/rooms/:id                    { ceiling_height_cm? } (기존 확장)
POST   /api/rooms/:id/features           { kind, wall, offset_cm, ... }
PATCH  /api/features/:id                 부분 수정
DELETE /api/features/:id
GET    /api/layout                       응답에 방별 features 포함 (기존 확장)
```

## 6. 단계 3 — 사다리차 / 창문 반입 (개요)

- 데이터: `home_settings`에 층수·사다리차 이용 가능 여부. 창 치수는 단계 2의 `room_features`에서 확보됨.
- 판정: 정상 동선(현관·엘베·방문) 불가 **그리고** 사다리차 가능 **그리고** 어느 창문이든 통과 가능 → "사다리차 반입 가능" 상태 추가.
- 자체 스펙은 착수 시점에 작성.

## 7. 단계 4 — 배치 현실성 (개요)

- 문 열림 부채꼴과 가구 간섭 경고, 가구 겹침·벽 이탈 감지, 통행 여유(60~90cm) 체크.
- 콘센트 위치를 앵커로 가전(TV·냉장고 등) 배치 가이드.
- 배치 좌표 기반 방별 천장 정밀 비교(5.5의 정교화).

## 8. 비범위

- "문에서부터 몇 cm" 편의 입력 (모서리 기준으로 통일)
- 캔버스 클릭/드래그로 부착물 배치 (폼 입력만)
- 대각선 기울이기 반입 판정 (보수적 판정 유지)
- 반입 경로(방까지 가는 동선) 모델링 — 대표 최소 문으로 근사
- 비사각형 방, 곡선 벽 (기존 스펙과 동일)

## 9. 에러 처리

- 검증 위반(5.4) 시 400 + 한글 메시지, 프론트는 기존 패턴대로 인라인 오류 표시.
- 없는 room/feature는 404. 부착물이 있는 방 삭제 시 CASCADE로 함께 삭제.

## 10. 테스트 전략

- **단계 1:** `carryin.js` 단위 테스트 확장 — 방문 통과/빠듯/불가, 방문 치수 미입력 시 체크 제외. `home-settings` API 필드 추가 검증.
- **단계 2:** `room_features` API 통합 테스트(정상/검증 실패/404), offset+폭 검증 단위 테스트, 기호 좌표 계산(벽 앵커 → SVG 좌표) 순수 함수 단위 테스트, 패널 ＋ 흐름 컴포넌트 테스트.

## 11. 구현 순서

1. 단계 1 (마이그레이션 → `carryin.js` TDD → `CarryInPage` 폼)
